2. Guida tecnica

Engine del MUD
Il MUD usa un breve script Perl driver.pl
Questo script utilizza in primo luogo il modulo MudDrv.pm. Si tratta di un oggetto specializzato che funge da motore del MUD e che utilizza le librerie standard IO::Socket, IO::Select per porre in essere essenzialmente:
    - un listner sulla porta specificata nella configurazione, per accettare nuove connessioni;
    - un selector per accettare gli input di tutti gli utenti connessi e gestire le risposte.
    - un gestore di oggetti che interagiscono tra di loro e con gli utenti connessi.
Il file di configurazione stabilisce i parametri vitali per il corretto funzionamento del driver stesso, come la struttura delle directory, la porta tcp alla quale connettersi, i vari timeout, l'elenco degli indirizzi-pirata bannati, ecc.

Oltre al file di configurazione iniziale (muddrv.cfg), che contiene informazioni basilari per il driver, sono presenti altri file di set-up; lo scopo di questi è di consentire una rapida modifica dell'aspetto del MUD, come le traduzioni in italiano o in un'altra lingua di tutti i messaggi emessi dal driver senza necessariamente dover fermare e riavviare il driver; cioè è possibile obbligare il driver a rileggere i file di configurazione in qualunque momento in modo che le modifiche vengano recepite immediatamente, se possibile.

Tranne il file di configurazione iniziale, tutti gli altri sono messi sotto la directory bin/setup; comunque anche la struttura delle directory è parametrizzabile sul primo file di configurazione.
  - constants.cfg consente di modificare i parametri di base, la lista dei canali di chat, i colori, il calendario, il tempo atmosferico, i livelli di avanzamento, i nomi "banditi" ecc. Essenzialmente, ogniqualvolta c'è bisogno di un "valore costante" ho pensato di fornire il luogo centralizzato dove poterle modificare con comodità.
  - messages.cfg consente di modificare qualunque messaggio "standard" del driver: per esempio il messaggio associato alla chiave "PromptLogin" manda il messaggio "Chi sei?" al terminale dell'utente appena si collega.
Per poter avviare un "nuovo" MUD è necessario mettere mano a questi due file.

Nel corso del tempo sono spuntati anche altri file di set-up, del tutto simili ai precedenti due ma specializzati nel loro particolare scopo.
  - actions.cfg consente di ridefinire i comandi di base: per esempio quando l'utente dà il comando "aiuto", in realtà viene invocato la subroutine "cmd_help" presente nel file "_help.pl" da qualche parte tra i moduli del MUD; in questo caso la definizione nel file di configurazione sarà qualcosa di simile a  aiuto = cmd::ghost::_help::cmd_help
  - emotes.cfg è la lista strutturata delle "emote" riconosciute nel MUD; per esempio il comando pacca permette di salutare, sorridere, dare una sonora pacca sulla spalla a qualcuno (ma anche sulla testa o sul didietro, ecc.) oppure di produrre altri tipi di manifestazioni personali.
  - adverbs.cfg è la lista degli avverbi riconosciuti nell'ambito delle "emote"; è possibile caratterizzare il modo con il quale una certa "emote" si manifesta: per esempio dare il comando pacca gino forte testa dovrebbe dare una pacca bella forte sulla testa di Gino, chiunque esso sia...
Questi tre file di configurazione non dovrebbero essere modificati molto, a meno che non si debba effettuare una "traduzione" in una lingua straniera; si può notare come il file contenga due serie di definizioni, quella in Inglese e quella in Italiano: dato che l'italiano è in coda al file, e viene letto per ultimo, prevale rispetto alla prima definizione. Tutto questo vorrebbe dare una parvenza di internazionalizzazione del codice che tendenzialmente utilizza nomi in inglese per le parti relative al "cuore" del codice e nomi in italiano per le parti più vicine all'utente.

Archivio dati - SQLite
Il driver impiega un database SQLite per garantire la persistenza di buona parte dei dati, ma per semplicità di gestione molti altri dati sono mantenuti in alcuni semplici file di configurazione; tipico il caso del file di configurazione del singolo utente.

La scelta di usare SQLite è avvenuta in modo molto semplice: la versione Perl 5.010 scaricata da ActiveState contiene già la libreria SQLite, senza dover fare la fatica di scaricare e installare altri pezzi. La versione qui disponibile è SQLite 3.3.7.

Per l'amministrazione del database ho utilizzato il plug-in di Firefox, semplice, intuitiva, efficace. In alternativa si può utilizzare uno script Perl sviluppato dal sottoscritto che fa da interfaccia testuale, estremamente spartana ma efficace; in alternativa, per Windows, si può usare l'interfaccia sqlite3.exe scaricabile dal sito di {SQLite}. Per il fatto che l'intero MUD è scritto in Perl ed e programmabile in Perl, per gli amministratori del MUD è disponibile il comando "sql" che consente di fare quasi le medesime cose che lo script appena citato.

Nel database sono definite inizialmente le seguenti tabelle
  - muddrv_channel_buffer contenuto dei "canali chat" 
  - muddrv_channel_subscriber partecipanti ai "canali chat"
  - muddrv_mailbox contenuto delle caselle di posta interna
  - muddrv_passwords criptate 
  - muddrv_room_annotation mantiene le annotazioni fatte nelle stanze
  - muddrv_standard_level tabella dei valori standard di ogni livello e razza 
  - muddrv_stock_exchange tabella dei valori dei beni in circolazione sul mercato
  - muddrv_user_info informazioni statistiche di ciascun utente

Funzionalità di base - Libreria
Le funzionalità di base, di utilità, il nucleo delle librerie è reso disponibile in un unico grande package MudCore.pm suddivise per categoria:
  - Utility: manipolazioni di stringhe, codici escape ANSI, ricerca, conversioni.
  - File: gestione di file, nome di file, navigazione directory, log
  - Config: gestione di file di configurazione (lettura e scrittura di oggetti)
  - User-Client related: salva, chiudi
  - Core: interazione con il driver, le connessioni, i salvataggi ecc.
  - Parsing e I/O: interazione con i client
  - Object Interface Library: interazione con gli oggetti del MUD

Componenti aggiuntivi - Daemon
Le funzionalità che trasformano un banale "engine" in un affascinante MUD sono racchiusi in alcuni componenti aggiuntivi che ho affettuosamente chiamato "daemon".

Questa è la lista dei daemon presenti
  - actions: Talvolta è utile maneggiare un comando come un oggetto statico.
  - channel: Gestione dei canali di chat interna.
  - combat: Gestione del combattimento dall'inizio fino alla vittoria o alla sconfitta 
  - death: Gestione del decesso di un personaggio.
  - emote: Gestione delle "emozioni" 
  - hazards: Gestione degli eventi rischiosi e catastrofici
  - level: Gestione dei livelli dei personaggi
  - mail: Gestiione della posta "elettronica" interna al MUD
  - move_dir: Gestione del movimento 
  - patch: Gestione delle patch sui profili
  - room_desc: Gestione della descrizione delle stanze
  - stock: Gestione dei prezzi degli oggetti presenti sul mercato
  - time: Gestione del tempo, stagioni, clima.
